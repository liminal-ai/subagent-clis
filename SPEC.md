# subagent-clis — cursor-sub SPEC (v1)

Repo that will eventually hold one small CLI per coding-agent backend (cursor, copilot, codex),
each presenting the same agent-ergonomic verbs and file contract. **This spec covers only the
first CLI: `cursor-sub`, wrapping the local `cursor-agent` binary.** Do NOT build a shared core
package yet — single self-contained package, but keep internal modules clean so a core can be
extracted later.

## Design philosophy (binding)

- The CLI is a thin set of conveniences over an open, documented file substrate. The file layout
  and event schema are the API; verbs are sugar for the hot path.
- Callers are AI agents. Output must be machine-readable JSON by default, quiet, and cheap to
  poll. No spinners, no ANSI, no progress noise on stdout.
- Never swallow errors: subprocess stderr goes to a file whose path is always reported.
- Wrap the vendor CLI (`cursor-agent`), not any SDK/API. Auth, harness, tools stay Cursor's job.
- Unknown flags pass through to `cursor-agent` verbatim. Do not build an abstraction over
  Cursor's flag surface.

## Repo / package layout

- pnpm workspace at repo root (`pnpm-workspace.yaml`), package at `packages/cursor-sub`.
- TypeScript, ESM, Node >= 20. Compile with `tsc` to `dist/`. Bin entry: `cursor-sub`.
- Dependencies: zero runtime deps preferred; `commander` allowed if it genuinely helps.
  Dev deps: typescript, vitest, @types/node. No linter setup in v1.
- Root README.md (short) + packages/cursor-sub/README.md (usage) + packages/cursor-sub/SCHEMA.md
  (the file/event contract — this doc is the API and must be complete and versioned).

## Session storage (the substrate)

Every run gets a directory: `~/.subagent-clis/cursor/sessions/<run_id>/`

- `run_id`: generated by cursor-sub at start: `<yyyymmdd-hhmmss>-<6 hex chars>`. This is the
  primary handle. Cursor's own chat/session id is captured separately for `--resume`.
- Files:
  - `meta.json` — written at spawn: `{schema_version, run_id, backend:"cursor", cwd, model,
    argv (full cursor-agent argv), prompt, started_at}`
  - `raw.jsonl` — verbatim cursor-agent `stream-json` events, appended as received
  - `stream.jsonl` — canonical events mapped from raw (see SCHEMA.md section below)
  - `stderr.log` — cursor-agent stderr
  - `pid` — pid of the runner process while running (do not delete on exit; liveness is checked,
    stale pidfile is fine once envelope exists)
  - `envelope.json` — written once on completion (see envelope)

## Envelope (result contract)

`envelope.json`, also what `exec`/`result` print on stdout:

```json
{
  "schema_version": 1,
  "backend": "cursor",
  "run_id": "20260701-153000-a1b2c3",
  "session_id": "<cursor's own session/chat id, for --resume>",
  "model": "composer-2.5",
  "cwd": "/path/where/it/ran",
  "status": "ok | error",
  "exit_code": 0,
  "result": "<final assistant message text>",
  "usage": { "...whatever cursor reports, verbatim, or null": true },
  "duration_ms": 12345,
  "started_at": "ISO8601",
  "ended_at": "ISO8601",
  "stream_path": "/abs/path/stream.jsonl",
  "raw_path": "/abs/path/raw.jsonl",
  "stderr_path": "/abs/path/stderr.log"
}
```

## Canonical event schema (stream.jsonl)

Document fully in SCHEMA.md with `schema_version: 1`. One JSON object per line:

```
{ "t": "message",     "role": "assistant", "text": "..." }
{ "t": "reasoning",   "text": "..." }
{ "t": "tool_call",   "name": "...", "args": { } }
{ "t": "tool_result", "name": "...", "ok": true, "output": "...(truncate at 10KB, note truncation)" }
{ "t": "usage",       "data": { } }
{ "t": "lifecycle",   "event": "start | end", "data": { } }
```

Add `"ts"` (ISO8601, stamped at receipt) to every event. Unknown raw event types map to
`{ "t": "other", "raw_type": "...", "data": {...} }` — never dropped.

**Authoritative reference for cursor's raw stream-json event shapes:** read the existing proven
parser at `/Users/leemoore/.claude/skills/cursor-subagent/scripts/cursor-result` (bash+jq). Its jq
selectors encode the real event/field names (assistant messages, tool calls, session id, result,
usage). Derive the mapping from that script; do not guess field names.

## Verbs

All verbs print JSON to stdout (single object or JSONL) unless `--text` is given where noted.
Global: `--dir <sessions-root override>` (also env `CURSOR_SUB_HOME`).

1. `cursor-sub exec "<prompt>" [passthrough flags...]`
   Synchronous: start + wait + print envelope. `--text` prints only the final assistant message.
2. `cursor-sub start "<prompt>" [passthrough flags...]`
   Detached async: spawn the runner fully detached (setsid/unref; survives parent exit), print
   `{run_id, dir}` immediately, exit 0. Implementation: re-invoke own bin with a hidden
   `_runner <session-dir>` command via spawn(detached, stdio to files), unref.
3. `cursor-sub status [run_id]`
   Cheap poll — pid liveness + `wc -l`-style event count, no full JSONL parse:
   `{run_id, running: bool, events: n, last_event_ts, has_envelope: bool}`.
   No run_id → latest run (by dir mtime).
4. `cursor-sub result [run_id]`
   Print envelope.json. If still running: exit code 3, `{running: true}` on stdout.
   `--wait` blocks until done (poll 500ms), optional `--timeout <sec>` → exit 4 on timeout.
5. `cursor-sub last [run_id]` — final assistant message, plain text.
6. `cursor-sub messages [run_id]` — all assistant `message` events, plain text separated by `\n---\n`.
7. `cursor-sub tools [run_id]` — JSONL of `tool_call`/`tool_result` events.
8. `cursor-sub list [-n 10]` — recent runs: `{run_id, status, model, started_at, cwd, prompt: first 80 chars}` JSONL.

## Execution profile (exec/start)

Defaults appended unless caller already supplied them: `--model composer-2.5 --trust --force
--sandbox disabled`. Exception: if passthrough contains `--mode plan`, `--mode ask`, or `--plan`,
do NOT add `--force`/`--sandbox disabled` (still add `--trust` and model default).
Cursor invocation: `cursor-agent --print --output-format stream-json <prompt> <flags...>`
(check `cursor-result` for exact argv ordering that is known to work).

`--resume <cursor-session-id>` passes through; the new run gets its own run_id and dir, and the
envelope records the cursor session_id.

## Errors & exit codes

- 0 ok · 1 usage/CLI error · 2 subagent run failed (envelope has status:error + stderr tail) ·
  3 result-not-ready · 4 wait timeout.
- If `cursor-agent` is missing: exit 1 with a JSON error naming the install command.
- Runner must always write an envelope, even on crash/nonzero exit (status:"error",
  result = best-effort last message or "", include last 2KB of stderr in `error` field).

## Testing (keep light)

Vitest smoke tests using a **stub cursor-agent** (a small script in test fixtures that emits a
canned stream-json sequence, including a tool call and final message; PATH-prefixed in tests).
Cover: exec envelope correctness, start/status/result lifecycle, canonical mapping of the canned
events, list. Do NOT call the real cursor-agent in tests.
